# Loop Engineering 自动化流水线设计器

遵循 `04_ClaudeCode搭配Codex六种接线机制操作手册.md` 和 `05_Cyrus說AI视频_Claude自动化调教分析.md` 规范，设计标准化的 AI 自动化工作流。

## 核心思想

> 你不再给 AI 一遍遍写提示词，而是设计一个能自我驱动、直奔目标的智能循环。

## 何时使用

当用户说：
- "帮我设计一个自动化流程..."
- "这个任务能不能让 AI 自动跑..."
- "怎么用 Loop 来做这个..."
- "设计一个 XX 的自动化流水线..."

## 循环的六大核心组件

每个 Loop 设计都要明确以下组件：

| 组件 | 作用 | 常见实现方式 |
|------|------|------------|
| **Automations（自动触发）** | 循环的"心跳"，使系统持续运行 | Claude Code 循环机制、定时任务、Webhook 触发 |
| **Workspaces（工作区隔离）** | 确保并行循环互不干扰 | Git Worktree、独立分支、容器化环境 |
| **Skills（技能库）** | 提供标准化能力 | 项目规范文档、代码模板、最佳实践库 |
| **Connectors（连接器）** | 对接外部工具与系统 | GitHub、Jira、CI/CD、飞书 Bot、API |
| **Sub-agents（子代理）** | 分工协作，专人专责 | 代码生成代理、测试代理、评审代理 |
| **Memory（外部记忆）** | 持久化保存状态 | Git、运行日志、向量数据库、状态文件 |

## Loop 设计模板

### 1. 典型四步循环骨架

```markdown
## 循环流程

1. **感知**：自动读取状态/日志/输入，定位问题
2. **行动**：自主执行任务、修改代码、生成内容
3. **验证**：自动运行检查、测试、校验标准
4. **自愈**：成功则推进，失败则自动调整策略并重试
5. **人审**：关键节点由人类进行审查与确认
```

### 2. 与传统自动化的本质区别

- 传统自动化：执行**固定序列**指令，无自主决策能力
- Loop Engineering：以**可验证的目标**为导向，能自主调整路径，直到目标达成

### 3. 自动化流水线设计清单

#### 🔧 循环定义

```markdown
### 循环目标
> 一句话说明这个 Loop 要达成什么

### 成功标准（可量化）
- [ ] 所有测试通过
- [ ] 代码编译无错误
- [ ] 输出文件生成
- [ ] 关键指标达标：例如 覆盖率 > 80%
```

#### 🛡️ 停止条件（必须定义）

```markdown
### 停止条件（非常重要）

✅ 正常停止：
- 成功标准全部达成
- 用户手动终止

❌ 异常停止（防止死循环）：
- 连续重试 N 次仍失败
- 运行时间超过阈值（例如 30 分钟）
- 错误类型属于"不可自愈"类
- 用户触发停止信号
```

#### 🔄 重试策略

```markdown
### 重试机制

- 最大重试次数：3 次
- 重试间隔：指数退避（1s → 2s → 4s）
- 失败后行为：
  1. 记录失败原因
  2. 调整策略（换思路、简化目标）
  3. 超过阈值则停止并提醒人工
```

#### 📋 验证清单

```markdown
### 每轮验证

每一步执行后必须验证：
- [ ] 语法/格式检查通过
- [ ] 逻辑正确性检查
- [ ] 没有引入新的错误
- [ ] 输出符合预期格式
```

### 4. 四阶段演进路线

评估当前自动化程度，选择合适的阶段：

1. **Prompt Engineering**（手动提示）→ 人驱动
2. **Context Engineering**（上下文优化）→ 优化输入
3. **Harness Engineering**（工具集成）→ 工具驱动
4. **Loop Engineering**（自主循环）← **当前目标阶段**

### 5. 推荐组合方案

按复杂度选择合适的机制组合：

#### 入门稳定版（适合现在使用）

```markdown
Worktree 隔离
+ Codex / Claude 执行
+ 人工确认合并
```

#### 进阶安全版（项目变复杂后）

```markdown
Worktree 隔离
+ Codex 在执行分支写代码
+ Claude 在主分支规划和验收
+ Hooks 自动跑测试
+ 双模型互审
```

#### 高阶自动化版（流程成熟后）

```markdown
Sub-agent 或 MCP 服务
+ Hooks
+ Worktree
+ 自动提交
+ 早晨人工审查
```

### 6. 风险红线（永远不要自动化）

以下事情永远不要自动化，必须人工处理：

- ❌ 删除大量文件或数据
- ❌ 修改密钥、权限、支付配置
- ❌ 自动发布到生产环境
- ❌ 自动合并到主分支
- ❌ 自动上传私密文件
- ❌ 退款、投诉、法律相关决策
- ❌ 未经 review 的过夜改动直接采用

### 7. 执行前总检查清单

```markdown
## 启动前检查

- [ ] 当前任务是否已有项目文件夹
- [ ] 项目是否在 D:\Workspace 内
- [ ] 是否需要 Git 仓库
- [ ] 是否需要 Worktree 隔离
- [ ] 是否有明确的验证命令
- [ ] 是否明确允许修改的文件范围
- [ ] 是否明确失败后停止条件
- [ ] 是否需要双模型 review
- [ ] 是否需要把经验写入复利与踩坑日志
```

### 8. 过夜流水线设计模板

如果需要设计过夜自动执行的流程：

```markdown
## 过夜流水线设计

### 任务拆解
将大需求拆成 N 个可验证子任务，每个子任务：
- 有独立目标
- 有独立验证命令
- 失败不影响其他任务

### 执行规则
1. 创建独立 Worktree
2. 串行执行任务
3. 每个任务完成后运行验证命令
4. 验证通过才提交
5. 验证失败则停止并记录失败原因
6. 第二天人工 review

### 第二天验收清单
- [ ] 查看每个提交的 diff
- [ ] 验证关键功能是否正确
- [ ] 检查是否有过度修改
- [ ] 只合并真正符合目标的提交
- [ ] 将经验写入复利与踩坑日志
```

## Claude Code 与 Codex 协作机制

### 机制 1：Git Worktree 双工作区分工

```markdown
### 适用场景
- 希望 Claude 和 Codex 分工明确
- 希望改动物理隔离，避免互相污染
- 希望所有变更都能通过 diff 审查
- 希望失败时可以直接丢弃分支

### 操作流程
1. 在项目目录确认 Git 状态
2. 创建执行分支和 worktree：`git worktree add ../Project-codex -b codex/task-name`
3. 主项目目录：Claude 负责需求澄清、任务拆解、验收标准、review
4. Worktree 目录：Codex 负责代码修改、测试补充、文档更新、自检
5. Codex 完成后查看 diff：`git diff main...codex/task-name`
6. 运行测试
7. 合格后合并
8. 清理 worktree：`git worktree remove ../Project-codex`
```

### 机制 2：双模型互审机制

```markdown
### 适用场景
- 希望减少单一模型盲区
- 希望一个模型写，另一个模型审
- 希望发现过度设计、漏测、边界条件、文件放错等问题

### review 模板
请只审查当前 diff，不要评价无关代码。
优先找 bug、回归风险、安全问题、测试缺口、文件位置错误。
按 P0/P1/P2/P3 排序。
每条问题必须包含文件路径、具体位置、原因和建议修复方式。
如果没有问题，请明确说明没有发现阻塞项，并指出剩余风险。

### 操作流程
1. Codex 执行具体改动
2. Claude Code 审查 diff
3. Claude 输出：严重问题 / 中等问题 / 小问题 / 测试缺口 / 是否建议合并
4. Codex 根据审查意见修复
5. Claude 复审
6. 通过后合并或交付
```

### 机制 3：Hooks 钩子自动触发执行

```markdown
### 适用场景
- 希望 Claude 写完代码后自动跑测试
- 希望每次编辑后自动检查格式、lint、类型
- 希望自动检查文件是否放在正确项目目录
- 希望减少"忘记验证"的问题

### 建议
先让 Hook 只提示，不阻断。稳定后再考虑阻断危险操作。
```

## 输出要求

最终交付的 Loop 设计方案必须包含：

1. ✅ 循环目标与成功标准
2. ✅ 六大组件配置清单
3. ✅ 四步循环详细流程
4. ✅ 明确的停止条件（防止死循环）
5. ✅ 重试策略与失败处理
6. ✅ 每轮验证清单
7. ✅ 风险红线与禁止项
8. ✅ 推荐的协作机制组合
9. ✅ 执行前检查清单
10. ✅ 分阶段实施路线

## 最值得记住的原则

> 先隔离，再自动化。先手动跑通，再写脚本。先能回滚，再放大执行。先沉淀规则，再追求无人值守。

> 规则正确 > 流程稳定 > 小范围试跑 > 自动化放大
