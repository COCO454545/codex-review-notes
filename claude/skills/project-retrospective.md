# 项目复盘与经验沉淀器

遵循 `01_全局复利与踩坑日志.md` 规范，对完成的项目进行标准化复盘，将经验转化为可复用的规则。

## 何时使用

当用户说：
- "这个项目做完了，帮我复盘一下"
- "把这次的经验沉淀下来"
- "更新一下复利与踩坑日志"
- "总结一下这次踩了哪些坑"
- "这次有什么可以复用的方法"

## 复盘核心原则

> 每一次错误和成功都应该沉淀为下一次可复用的经验。

### 3. 踩坑记录（最重要）

使用标准模板记录每个问题：

```markdown
## 踩坑记录

### 坑 001：问题简要描述

**发生时间**：
**关联文件/模块**：

**发生了什么**：
> 详细描述问题现象，越具体越好

**根本原因**：
> 深入分析 5 个为什么，找到根源

**造成的影响**：
- 影响1：浪费了 X 时间
- 影响2：需要返工 Y
- 影响3：...

**解决方法**：
> 记录最终是怎么解决的，步骤要具体

**未来预防**：
> 下次怎么避免再踩这个坑
- [ ] 预防措施1
- [ ] 预防措施2

**可复用方法**：
> 这个解决方法能不能变成通用规则？
- 可以/不可以
- 如果可以，怎么复用？

**是否需要更新 AGENTS.md 或工作区提示词**：是/否
```

每个坑都要按这个模板记录，不要怕麻烦。

## 常见问题分类参考

复盘时可以按这些分类整理：

| 分类 | 说明 |
|------|------|
| 重复纠错 | 同样的问题出现多次 |
| 需求理解错误 | 一开始就理解错了需求 |
| 文件管理混乱 | 文件放错位置、找不到 |
| 项目目录创建不规范 | 没有按 SOP 创建 |
| 产出文件散落到错误位置 | 违反归档规则 |
| 全盘扫描浪费时间 | 没有先定位再操作 |
| 过度设计 | 做了不需要的功能 |
| 未验证结果 | 改完就交，没验证 |
| 交付物和过程文件混放 | 目录用错了 |
| 没有及时复盘 | 做完就忘，经验流失 |
| AI 生成结果未审核 | 过度信任 AI |
| 没有设置停止条件 | 死循环浪费时间 |

## 输出要求

完整的复盘报告必须包含：

1. ✅ 项目基本信息与交付成果
2. ✅ 目标达成情况评估
3. ✅ 至少 3 条踩坑记录（按标准模板）
4. ✅ 至少 2 条优质做法记录（按标准模板）
5. ✅ 可复用提示词/模板提取
6. ✅ 时间投入分析与效率优化建议
7. ✅ 工具/方法效果评估
8. ✅ 全局文档更新建议
9. ✅ 下次改进行动计划
10. ✅ 一句话总结

## 关键提醒

- 🎯 **复盘不是为了追责，而是为了不重复踩同样的坑**
- 💡 **同样的坑踩第三次，就是自己的问题了**
- 🔄 **优质做法要主动提炼成可复用的规则，而不只是"这次运气好"**
- 📝 **能跨项目复用的经验，一定要同步到全局复利与踩坑日志**
- ⚠️ **如果发现需要更新 AGENTS.md 或工作区提示词，不要犹豫，马上提**
