# Claude Code 搭配 Codex 六种接线机制操作手册

本文档用于帮助理解 Claude Code 与 Codex 的 6 种高阶组合方式。重点不是一次性全部搭建，而是判断每种机制适合什么场景、需要做哪些任务、操作流程是什么、有哪些风险。

## 总体建议

推荐落地顺序：

1. Git Worktree 双工作区分工
2. 双模型互审机制
3. Hooks 钩子自动触发执行
4. Codex 封装为 MCP 服务接入
5. Codex 作为 Claude Code 子代理
6. 过夜自动化流水线

核心原则：

- 先隔离，再自动化。
- 先手动跑通，再写脚本。
- 先能回滚，再放大执行。
- 先沉淀规则，再追求无人值守。

## 机制 1：Codex 作为子代理 Sub-Agent

### 适用场景

- 希望在 Claude Code 中一句话调度 Codex。
- Claude 负责拆任务、规划和总结。
- Codex 在后台执行某类固定任务，例如补测试、重构小模块、生成文档。

### 前置条件

- 已安装 Claude Code。
- 已安装并能命令行调用 Codex。
- 清楚 Codex CLI 或 API 的调用方式。
- 明确子代理能做什么、不能做什么。

### 任务清单

- 确认 Codex 可在终端独立运行。
- 确认 Claude Code 的 subagent 配置目录。
- 编写一个专用 subagent，例如 `codex-worker`。
- 限制 subagent 权限，优先只给必要工具。
- 设计 Codex 调用脚本。
- 设计输出格式，例如只返回总结、diff 路径、测试结果。
- 小任务试跑。
- 将成功经验写入复利与踩坑日志。

### 操作流程

1. 在 Claude Code 中创建自定义 subagent。
2. 为 subagent 写清楚职责：
   - 只接收明确任务。
   - 不做需求扩展。
   - 执行前确认目标文件。
   - 执行后返回改动摘要和验证结果。
3. 让 subagent 调用本地 Codex 命令或封装脚本。
4. 用一个低风险任务测试，例如“阅读某个文件并给出修改建议”。
5. 再测试写文件任务。
6. 最后再允许它执行测试、生成补丁或提交。

### 注意事项

- 不建议一开始让子代理拥有全权限。
- 子代理会增加上下文和调用成本。
- 子代理内部再调用 Codex，调试链路会变长。
- 如果输出格式不固定，Claude 很难稳定接收结果。
- 适合固定重复任务，不适合临时探索任务。

### 推荐级别

中后期再做。当前优先级不高。

## 机制 2：Codex 封装为 MCP 服务接入

### 适用场景

- 希望 Claude Code 像调用工具一样调用 Codex。
- 需要参数、上下文、返回值都结构化。
- 希望未来接入多个工具，形成统一工具协议。

### 前置条件

- 理解 MCP 的基本概念。
- 能写一个本地 stdio 或 HTTP MCP 服务。
- 已明确 Codex 的输入输出边界。
- 能处理权限、日志和错误返回。

### 任务清单

- 定义 MCP 工具名称，例如 `codex_run_task`。
- 定义输入参数：
  - `project_path`
  - `task`
  - `allowed_files`
  - `mode`
  - `verification_command`
- 定义输出结构：
  - `summary`
  - `changed_files`
  - `verification_result`
  - `warnings`
- 编写 MCP 服务。
- 在 Claude Code 中注册 MCP 服务。
- 测试只读任务。
- 测试小范围写入任务。
- 增加日志和失败回退机制。

### 操作流程

1. 先写最小 MCP 服务，只支持一个只读工具。
2. 让 Claude Code 调用该工具读取项目状态或生成计划。
3. 确认返回结构稳定。
4. 再增加 Codex 执行能力。
5. 给工具加安全边界：
   - 限定项目路径。
   - 限定可写文件。
   - 禁止删除文件。
   - 禁止访问 Workspace 外部路径。
6. 小项目试跑。
7. 稳定后再扩展更多工具。

### 注意事项

- MCP 能力强，但安全风险也高。
- 不要把用户输入直接拼进 shell 命令。
- 不要让 MCP 服务拥有无限文件权限。
- 不要先做复杂功能，先做最小可验证版本。
- 对外部网页、仓库、文档内容要防 prompt injection。

### 推荐级别

中后期可做。适合你已有稳定工作流后再升级。

## 机制 3：Git Worktree 双工作区分工

### 适用场景

- 希望 Claude 和 Codex 分工明确。
- 希望改动物理隔离，避免互相污染。
- 希望所有变更都能通过 diff 审查。
- 希望失败时可以直接丢弃分支。

### 前置条件

- 项目必须是 Git 仓库。
- 已有可运行的测试或检查命令。
- 清楚主分支和执行分支的关系。

### 任务清单

- 确认项目在 `D:\Workspace\Project_编号_项目名称` 下。
- 初始化或确认 Git 仓库。
- 确认当前主工作区干净。
- 创建 worktree 分支。
- 在 worktree 中让 Codex 执行任务。
- 在主工作区让 Claude 负责计划和验收。
- 查看 diff。
- 运行测试。
- 决定合并、修改或丢弃。

### 操作流程

1. 在项目目录确认 Git 状态。

```bash
git status
```

2. 创建执行分支和 worktree。

```bash
git worktree add ../Project_001_项目名称-codex -b codex/task-name
```

3. 在主项目目录中让 Claude 做：
   - 需求澄清
   - 任务拆解
   - 验收标准
   - review

4. 在 worktree 目录中让 Codex 做：
   - 代码修改
   - 测试补充
   - 文档更新
   - 自检

5. Codex 完成后查看 diff。

```bash
git diff main...codex/task-name
```

6. 运行测试。

```bash
npm test
```

或按项目实际命令执行。

7. 合格后合并。

```bash
git merge codex/task-name
```

8. 清理 worktree。

```bash
git worktree remove ../Project_001_项目名称-codex
```

### 注意事项

- 不要在两个工作区同时改同一批文件。
- `.env`、密钥、本地配置通常不会自动复制到 worktree。
- worktree 路径也要放在 `D:\Workspace` 内，避免文件散落。
- 合并前必须看 diff。
- 不要让 Codex 直接改主分支。

### 推荐级别

最高。建议优先掌握。

## 机制 4：Hooks 钩子自动触发执行

### 适用场景

- 希望 Claude 写完代码后自动跑测试。
- 希望每次编辑后自动检查格式、lint、类型。
- 希望自动检查文件是否放在正确项目目录。
- 希望减少“忘记验证”的问题。

### 前置条件

- 已能手动运行测试或检查命令。
- 项目有稳定的命令，例如 `npm test`、`npm run lint`。
- 明确哪些事件触发 Hook。
- 明确 Hook 失败时如何处理。

### 任务清单

- 选定一个最小 Hook 场景。
- 写一个检查脚本。
- 手动运行脚本，确认稳定。
- 配置 Claude Code Hook。
- 先让 Hook 只提示，不阻断。
- 稳定后再考虑阻断危险操作。
- 记录误报和漏报。

### 操作流程

1. 先选一个最小目标，例如“编辑后自动跑 lint”。
2. 写一个本地脚本，例如：

```bash
npm run lint
```

3. 手动运行，确认不会太慢。
4. 配置 Claude Code 的 `PostToolUse` Hook。
5. 限定触发范围，例如只在写文件后触发。
6. 观察几次真实任务。
7. 如果 Hook 太慢，改成只检查变更文件。
8. 如果 Hook 误报，先记录，不要急着扩大规则。

### 注意事项

- Hook 是确定性控制层，适合做检查，不适合做复杂决策。
- 不要一开始配置太多 Hook。
- 慢 Hook 会严重影响体验。
- 自动修复类 Hook 风险高，建议先只做检查。
- 涉及删除、移动、提交、上传时必须保守。

### 推荐级别

高。建议在 Worktree 跑顺后加入。

## 机制 5：双模型互审机制

### 适用场景

- 希望减少单一模型盲区。
- 希望一个模型写，另一个模型审。
- 希望发现过度设计、漏测、边界条件、文件放错等问题。

### 前置条件

- 有明确的审查标准。
- 有 diff 或具体文件可审。
- 有测试或人工验收标准。

### 任务清单

- 指定主执行模型。
- 指定复审模型。
- 编写固定 review 模板。
- 每次只审当前改动，不扩散到无关文件。
- 要求审查结果按严重程度排序。
- 要求只提出可执行问题。
- 审查后决定采纳或忽略。
- 将重复问题写入复利与踩坑日志。

### 操作流程

1. Codex 执行具体改动。
2. Claude Code 审查 diff。
3. Claude 输出：
   - 严重问题
   - 中等问题
   - 小问题
   - 测试缺口
   - 是否建议合并
4. Codex 根据审查意见修复。
5. Claude 复审。
6. 通过后合并或交付。

### 推荐 review 模板

```text
请只审查当前 diff，不要评价无关代码。
优先找 bug、回归风险、安全问题、测试缺口、文件位置错误。
按 P0/P1/P2/P3 排序。
每条问题必须包含文件路径、具体位置、原因和建议修复方式。
如果没有问题，请明确说明没有发现阻塞项，并指出剩余风险。
```

### 注意事项

- 互审不是让两个模型互相聊天，而是围绕 diff 验收。
- 不要把 review 变成无关重构建议。
- 审查意见必须能落地。
- 如果两个模型意见冲突，以测试、需求和用户目标为准。
- 重复出现的问题要沉淀为规则。

### 推荐级别

很高。现在就可以手动使用。

## 机制 6：过夜自动化流水线

### 适用场景

- 需求已拆成多个清晰子任务。
- 每个子任务都有验证命令。
- 项目测试覆盖较好。
- 有 Git 分支隔离和自动提交规则。
- 第二天愿意逐个审查 diff。

### 前置条件

- 项目必须在 Git 仓库中。
- 必须使用独立分支或 worktree。
- 必须有测试命令。
- 必须有自动提交规范。
- 必须有失败停止规则。

### 任务清单

- 将大需求拆成可验证子任务。
- 为每个子任务写成功标准。
- 为每个子任务写验证命令。
- 设置 Codex Full Auto 或等价自动模式。
- 每完成一个子任务自动提交。
- 失败时停止，不继续扩散。
- 第二天人工 review。
- 合格才合并。

### 操作流程

1. 睡前整理任务列表。

```text
任务 1：
目标：
允许修改：
验证命令：
成功标准：

任务 2：
目标：
允许修改：
验证命令：
成功标准：
```

2. 创建独立 worktree。
3. 让 Codex 串行执行任务。
4. 每个任务完成后运行验证命令。
5. 验证通过才提交。
6. 验证失败则停止并记录失败原因。
7. 第二天查看提交记录。
8. 对每个提交进行 review。
9. 只合并真正符合目标的提交。

### 注意事项

- 不要在主分支过夜自动执行。
- 不要让它无限制改全项目。
- 不要允许失败后继续乱试。
- 不要自动合并。
- 不要自动发布。
- 不要处理高风险任务，例如支付、权限、数据删除、安全配置。
- 第二天必须人工验收。

### 推荐级别

低到中。只适合系统成熟后使用。

## 六种机制对比表

| 机制 | 价值 | 成本 | 风险 | 当前建议 |
|---|---|---:|---:|---|
| Sub-Agent | 统一入口、后台执行 | 高 | 中 | 后做 |
| MCP | 结构化工具调用 | 高 | 高 | 后做 |
| Worktree | 文件隔离、可回滚 | 中 | 低 | 优先 |
| Hooks | 自动检查、减少漏验 | 中 | 中 | 第二阶段 |
| 双模型互审 | 降低盲区 | 低 | 低 | 立即使用 |
| 过夜流水线 | 放大产能 | 高 | 高 | 成熟后 |

## 推荐组合方案

### 入门稳定版

适合现在使用：

```text
D:\Workspace 全局规则
+ Project_编号_项目名称
+ Codex 执行
+ Claude/Codex 互审
+ 人工确认合并
```

### 进阶安全版

适合项目开始变复杂后使用：

```text
Worktree 隔离
+ Codex 在执行分支写代码
+ Claude 在主分支规划和验收
+ Hooks 自动跑测试
+ 双模型互审
```

### 高阶自动化版

适合流程成熟后使用：

```text
Sub-Agent 或 MCP
+ Hooks
+ Worktree
+ 自动提交
+ 人工早晨审查
```

## 执行前总检查清单

- 当前任务是否已有项目文件夹。
- 项目是否在 `D:\Workspace` 内。
- 是否需要 Git 仓库。
- 是否需要 worktree。
- 是否有验证命令。
- 是否明确允许修改的文件范围。
- 是否明确失败后停止条件。
- 是否需要双模型 review。
- 是否需要把经验写入复利与踩坑日志。

## 风险红线

以下事情不要自动化：

- 删除大量文件。
- 移动 Workspace 外部文件。
- 修改密钥、权限、支付、账号配置。
- 自动发布生产环境。
- 自动合并到主分支。
- 自动上传私密文件。
- 未经 review 的过夜改动直接采用。

## 学习路线

建议按这个顺序练习：

1. 手动双模型互审一次。
2. 手动创建一个 worktree，让 Codex 在里面改小任务。
3. 主分支查看 diff 并验收。
4. 加一个最小 Hook 自动跑测试。
5. 写一个固定 review 模板。
6. 跑一个半自动多步骤任务。
7. 最后再研究 MCP 或 Sub-Agent。
