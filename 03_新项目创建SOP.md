# 03_新项目创建SOP

> 本 SOP 用于规范在 `D:\Workspace` 下创建、维护和归档普通项目。它服务的是**项目层**，而不是全局沉淀层。

## 一、什么时候必须创建项目

出现以下任一情况，都应创建项目文件夹：

1. 用户要求做新的工具、应用、视频、文档、自动化、调研、实验或交付物。
2. 当前任务不属于任何已有 `Project_编号_项目名称` 文件夹。
3. 任务会产生多个文件、素材、脚本、截图、输出或过程记录。
4. 任务需要后续维护、持续迭代、复盘或验收记录。
5. 任务会形成独立的业务资产，而不是只沉淀规则。

以下情况**不创建项目**：

1. 简单问答、翻译、解释。
2. 一次性命令查询。
3. 不产生文件、不需要后续维护的小任务。
4. Workspace 级别的规则沉淀、偏好沉淀、AGENTS/SOP/skills 调整。
5. GitHub 同步规范、全局复盘、跨电脑复用说明。

判断原则：

- 产生“业务交付物”或“项目资产” → 创建 `Project_*`
- 产生“全局规则、偏好、SOP、方法论、skills、同步规范” → 直接更新根目录沉淀层，不创建项目

## 二、项目命名规范

项目文件夹统一命名：

```text
Project_编号_项目名称
```

当前统一规则：
1. 编号使用两位数字，如 `01`、`02`、`03`。
2. 新项目取当前最大编号加一。
3. `Project_00_全局工作台` 为保留目录，不参与普通项目编号。
4. 项目名称要短、清晰、可搜索。
5. 可使用中文、英文、数字、下划线。
6. 不使用空格、斜杠、冒号、问号、星号等特殊符号。

## 三、项目标准目录结构

```text
Project_编号_项目名称
├─ 00_project_brief.md
├─ 01_requirements
├─ 02_references
├─ 03_assets
├─ 04_src
├─ 05_outputs
├─ 06_tests
├─ 07_logs
├─ 08_review
└─ 99_temp
```

### 目录用途
- `00_project_brief.md`：项目目标、范围、当前状态、关键文件、下一步。
- `01_requirements`：需求、任务说明、验收标准、变更记录。
- `02_references`：参考资料、截图、网页、竞品、来源说明。
- `03_assets`：图片、音频、视频、字体、Logo、原始素材。
- `04_src`：代码、脚本、配置、工程文件。
- `05_outputs`：最终交付物、导出文件、发布版本。
- `06_tests`：测试记录、截图、验证结果。
- `07_logs`：执行过程、关键命令、报错记录。
- `08_review`：复盘、反馈、改进建议。
- `99_temp`：短期临时文件，完成后清理或归档。

## 四、创建流程

### Step 1：确认项目编号

查看已有普通项目，取最大编号加一。`Project_00_全局工作台` 不计入普通项目编号。

### Step 2：创建项目文件夹和目录

创建：
- 项目目录
- 标准子目录
- `00_project_brief.md`

### Step 3：填写项目简报

`00_project_brief.md` 至少包含：

```text
# Project_编号_项目名称

## 项目目标

## 用户需求

## 当前状态

## 验收标准

## 目录说明

## 关键文件

## 下一步
```

### Step 4：更新全局项目索引

在 `02_全局项目索引.md` 中追加项目记录，至少填写：
- 编号
- 项目名称
- 文件夹
- 状态
- 项目目标
- 下一步
- 更新时间

### Step 5：开始执行

文件按类型放入对应目录：
- 需求、聊天摘要、任务说明 → `01_requirements`
- 参考图、参考网页、竞品资料 → `02_references`
- 图片、视频、音频、字体、Logo → `03_assets`
- 代码、脚本、配置、工程文件 → `04_src`
- 最终导出、交付版本、成品 → `05_outputs`
- 测试报告、截图、验证记录 → `06_tests`
- 执行日志、报错记录、命令记录 → `07_logs`
- 复盘、反馈、优化建议 → `08_review`
- 临时中转文件 → `99_temp`

## 五、旧版目录映射

如果从旧结构迁移资料，可按下列方式归入：

| 旧目录 | 新目录 |
| --- | --- |
| `00_项目说明.md` | `00_project_brief.md` |
| `01_需求与目标` | `01_requirements` |
| `02_素材与资料` | `02_references` 或 `03_assets` |
| `03_过程文件` | `04_src`、`06_tests`、`07_logs` 或 `99_temp` |
| `04_交付成果` | `05_outputs` |
| `05_复盘与日志` | `08_review` 或 `07_logs` |

## 六、外部文件处理规则

1. 用户提供的截图、图片、压缩包、文档等外部文件，如果会被项目使用，应复制或移动到项目目录。
2. 参考资料放入 `02_references`，原始素材放入 `03_assets`。
3. 如果文件体积很大且必须保留原位置，应在项目简报中记录原路径和用途。
4. 不依赖临时目录中的文件作为长期项目资产。

## 七、项目完成标准

项目完成前检查：

- [ ] 交付物已放入 `05_outputs`
- [ ] 源文件或脚本已放入 `04_src`
- [ ] 关键素材已放入 `03_assets` 或 `02_references`
- [ ] 测试或验证记录已放入 `06_tests`
- [ ] 重要过程日志已放入 `07_logs`
- [ ] 复盘内容已放入 `08_review`
- [ ] 可复用经验已同步到 `01_全局复利踩坑日志.md`
- [ ] 项目状态已更新到 `02_全局项目索引.md`
- [ ] `99_temp` 已清理或归档

## 八、禁止事项

1. 禁止把项目文件直接放在 Workspace 根目录。
2. 禁止把项目资料长期放到桌面、下载目录、临时目录或其它无关位置。
3. 禁止多个项目混用一个文件夹。
4. 禁止只输出最终文件而不保留关键过程资料。
5. 禁止把临时文件长期留在 `99_temp`。
6. 禁止把全局规则复盘误建成普通项目文件夹。
7. 禁止把项目层内容当成 GitHub 沉淀层直接同步。

## 九、和全局沉淀层的关系

本 SOP 只管理普通项目层。以下内容不属于项目层：
- 全局规则
- 工作区偏好
- 复利方法论
- GitHub 同步规则
- skills 体系

这些内容应直接沉淀到：
- Workspace 根目录 md
- `D:\Workspace\skills\`

而不是放进某个普通 `Project_*` 目录里。
