## 一句话结论

这个视频最值得学习的不是“全自动印钞厂”这个结果，而是背后的思路：

把 Claude 从聊天工具升级为“有规则、有工具、有流程、有复盘、有边界的自动化执行系统”。

## 可以学到的核心方法

### 1. 不要只调教模型，要搭建工作系统

普通用法：

```text
用户提问 -> Claude 回答 -> 用户手动执行
```

高级用法：

```text
目标输入 -> 规则读取 -> 任务拆解 -> 工具调用 -> 自动执行 -> 验证 -> 复盘沉淀
```

可迁移到当前 Workspace 的做法：

- 用 `AGENTS.md` 固定底层规则。
- 用 `00_全局控制台.md` 固定工作区总入口。
- 用 `01_全局复利与踩坑日志.md` 固定经验沉淀。
- 用 `03_新项目创建SOP.md` 固定项目创建和文件归档。

### 2. “印钞厂”的本质是可重复业务流程

不要理解成 AI 会自动赚钱。更准确的理解是：

```text
可重复需求 + 标准化输入 + 自动化执行 + 可验证输出 + 持续复盘
```

适合自动化的任务：

- 内容选题整理
- 文案初稿生成
- 素材归档
- 代码脚手架创建
- 数据清洗
- 测试生成
- 报告初稿
- 客户线索整理
- 固定格式交付物生成

不适合直接自动化的任务：

- 高风险账号操作
- 自动发布生产内容
- 自动付款
- 自动删除数据
- 需要强审美判断的最终交付
- 需要法律、财务、医疗责任判断的内容

### 3. Claude 调教的重点是“角色 + SOP + 工具 + 验收”

一个成熟 Claude/Codex 工作流至少包含：

```text
角色：你负责什么，不负责什么
输入：开始前必须读哪些文件
流程：按什么步骤执行
工具：可以调用哪些命令或服务
边界：哪些事情必须先问用户
验收：怎样判断任务完成
复盘：哪些经验要写回文档
```

当前 Workspace 已经具备基础结构：

- `AGENTS.md`：角色和底层行为规则
- `00_全局控制台.md`：总入口
- `01_全局复利与踩坑日志.md`：复盘沉淀
- `02_工作区提示词.md`：长期提示词
- `03_新项目创建SOP.md`：项目文件管理
- `04_ClaudeCode搭配Codex六种接线机制操作手册.md`：Claude Code + Codex 协作方案

### 4. 真正的自动化不是一次 prompt，而是流水线

建议的自动化流水线：

```text
用户输入目标
-> 创建 Project_编号_项目名
-> 生成项目说明
-> 拆解任务清单
-> 执行第一个子任务
-> 验证结果
-> 记录过程文件
-> 产出交付物
-> 写入复盘日志
-> 将可复用经验同步到全局文档
```

这比单纯“写一个神级提示词”更可靠。

## 对当前 Workspace 的启发

### 启发 1：新增“项目商业化验证”维度

如果未来目标是让 AI 辅助赚钱，每个项目说明里可以增加：

```text
商业目标：
目标用户：
交付物：
变现方式：
验证指标：
最小可行版本：
风险：
```

### 启发 2：每个项目都要有可验证产出

不要只让 AI “研究一下”。要让它产出明确文件：

- 需求文档
- 执行计划
- 代码或脚本
- 数据表
- 素材包
- 发布文案
- 复盘报告

### 启发 3：自动化前先定义停止条件

任何自动化任务都要写：

```text
什么情况下继续？
什么情况下停止？
什么情况下必须问用户？
什么情况下必须回滚？
```

这能避免 AI 在错误方向上越跑越远。

### 启发 4：把 Claude/Codex 当成“员工”，不是“神”

更合理的用法：

- 给岗位说明书。
- 给 SOP。
- 给文件位置。
- 给验收标准。
- 给复盘机制。

不合理的用法：

- 一句话要求它自动赚钱。
- 不给资料、不设边界、不验收。
- 让它直接操作账号、发布、付款或删除。

## 可落地的任务清单

### 第一阶段：补齐项目模板

- 在 `03_新项目创建SOP.md` 中加入商业化项目字段。
- 在新项目模板中加入“成功标准”和“验证指标”。
- 每个项目都要求有 `00_项目说明.md`。

### 第二阶段：建立自动化执行模板

新增一个可复用提示词：

```text
请基于当前项目目标，拆成 3-7 个可验证子任务。
每个子任务必须包含：目标、输入文件、输出文件、允许修改范围、验证方式、停止条件。
未经确认，不要执行高风险操作。
```

### 第三阶段：建立复盘闭环

每次完成任务后，固定检查：

- 本次是否节省了人工时间。
- 是否出现重复错误。
- 是否有可复用 SOP。
- 是否应该更新 AGENTS.md。
- 是否应该更新工作区提示词。

### 第四阶段：再考虑半自动流水线

等项目模板稳定后，再逐步使用：

- Git Worktree 隔离执行
- 双模型互审
- Hooks 自动测试
- MCP 或 Sub-Agent
- 过夜自动化流水线

## 需要警惕的地方

### 1. “全自动印钞厂”是营销表达

AI 可以提高效率，但不能保证赚钱。赚钱仍然取决于：

- 需求是否真实
- 渠道是否存在
- 产品是否有价值
- 成本是否可控
- 交付是否稳定

### 2. 自动化会放大错误

如果规则错了，自动化会更快地产生错误结果。

所以优先级应该是：

```text
规则正确 > 流程稳定 > 小范围试跑 > 自动化放大
```

### 3. 不要跳过验证

任何“自动赚钱系统”都必须有指标：

- 产出数量
- 转化率
- 成本
- 时间
- 错误率
- 人工复核成本

没有指标，就无法判断它是在赚钱还是制造噪音。

## 对现有规则的建议更新

建议后续把以下规则加入 `02_工作区提示词.md` 或新项目模板：

```text
当项目带有商业化、自动化、增长、变现目标时，必须先定义：
1. 目标用户
2. 交付物
3. 变现路径
4. 验证指标
5. 最小可行版本
6. 风险边界
7. 停止条件
```

## 最值得学的一句话

不要追求一个万能 prompt，而是搭建一个会持续沉淀经验、能自动执行、能验证结果、能及时停下来的 AI 工作系统。
