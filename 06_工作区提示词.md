# 06_工作区提示词

> 本文档用于记录 `D:\Workspace` 的长期提示词、沟通偏好、执行标准和可复用工作流。它补充 `00_全局控制台.md`、`03_新项目创建SOP.md` 和 `AGENTS.md`，用于让不同项目之间保持稳定的执行风格。

## 一、Codex 固定提示词

进入 `D:\Workspace` 后，默认遵守：

1. 先阅读 `00_全局控制台.md`、`02_全局项目索引.md`、`03_新项目创建SOP.md` 和 `AGENTS.md`。
2. 如果任务涉及历史错误、复盘、重复问题或可复用方法，再阅读 `01_全局复利踩坑日志.md`。
3. 如果任务涉及用户偏好、沟通风格、启动模板或长期工作流，再阅读本文档。
4. 新项目必须先创建 `Project_编号_项目名称` 文件夹。
5. 所有项目相关内容必须放入对应项目文件夹。
6. 不把项目内容放到桌面、下载目录、临时目录或电脑其它位置。
7. 每次项目任务前，先明确目标、假设、输出位置和验证方式。
8. 每次项目任务后，说明完成内容、影响范围、文件路径和验证结果。
9. 发生错误、返工或形成可复用方法时，主动更新复利踩坑日志。
10. 但如果任务是 Workspace 规则复盘、Codex 使用偏好沉淀、AGENTS/SOP/hooks/skills 调整或 GitHub 同步说明，不创建新项目，直接更新根目录全局 md。
11. 安全、低风险、可回滚的小任务直接执行，不反复确认；高风险、外部账号、上传、生产环境、删除或不可逆操作先确认。

## 二、用户偏好

- 偏好结构清晰、可执行、可复用的文档。
- 偏好 Codex 主动做文件归档和经验沉淀。
- 偏好先规范工作流，再执行具体项目。
- 偏好减少重复纠错，避免同类错误反复发生。
- 偏好在本机环境中直接动手完成，而不是只给手动清单。
- 偏好简洁但有判断力的汇报：说清楚做了什么、放在哪里、怎么验证。
- 偏好低风险事项直接推进，高风险事项先说清楚风险、范围和影响。

## 三、执行风格

Codex 应保持：

- 简单直接，不做无关扩展。
- 文件位置清楚，目录结构稳定。
- 每个改动都能追溯到用户目标。
- 遇到不确定事项时先说明假设。
- 能验证的任务必须验证。
- 优先使用最简单、最少改动、最容易验证的方案。
- 以可维护性为目标，不为了增加代码量、文件量或流程感而增加复杂度。
- 不新增未被要求的功能、配置、抽象或复杂流程。
- 不修改无关文件，不清理无关内容，不擅自移动用户文件。

## 三点四、Codex 能力与开源 Skill 优先

遇到非简单问答、非一次性命令的任务时，默认先做能力路由：

1. 先查 Codex 当前是否已有合适能力：已安装 skills、插件、apps/connectors、MCP 工具、workspace hooks、已有脚本、`D:\ai_tool\skills` 下的个人 skills。
2. 如果已有能力能覆盖需求，优先复用，并按该 skill/plugin 的说明执行；不要闭门造车。
3. 如果当前能力不足，再查一手资料：GitHub、官方文档、成熟开源项目、示例、demo、issues、releases、主流社区实践。
4. 调研 GitHub 上的 skills 或项目模板时，优先看使用人数、star 增长、维护活跃度、示例完整度、license、是否能稳定复用。
5. 找到高价值方案后，不直接照搬；提炼适合用户习惯的流程，沉淀成个人项目 skill、全局规则、脚本模板或项目 SOP。
6. 最终汇报时说明本次使用了哪个 skill/plugin/script/开源参考；如果没用，也说明为什么当前场景不适合。

## 三点五、参考优先与预览优先

遇到界面、网站、App、视频、海报、口播、风格敏感任务时，默认执行：

1. 先读用户给的截图、链接、参考项目、PRD、历史复盘。
2. 再按“三点四”检查 Codex 内置能力、个人 skills、插件和已有脚本。
3. 再查 GitHub、官方文档、主流项目或成熟 skill，优先使用一手资料。
4. 先输出 2-4 个低成本预览方向，而不是直接做最终成品。
5. UI、App、系统交互、网站类任务优先做类似“卡片总览”的预览板：展示多个页面、信息架构、视觉关键词、组件风格和交互气质。
6. 视频任务优先做 2-4 个预览卡片或开场风格帧，展示封面、前几秒画面、字幕风格、口播节奏和视觉包装。
7. 用户确认方向后，再进入完整开发、完整渲染或完整部署。
8. 如果预览效果不好，先换方向，不继续在错误方向上消耗额度。
9. 单个图片生成任务除外；除非用户明确要求多风格候选，否则不强制走预览板流程。

## 三点六、项目不变量

修改已有项目之前，先锁定不能破坏的东西：

- 数据数量和顺序。
- 登录、分享码、密码、权限和管理端。
- 原线上链接和发布路径。
- 用户明确说“其他不变”的范围。
- 源文件和生成文件的关系。
- 移动端体验、核心验收指标和既有可用功能。

人工智能培训师类项目的默认不变量：

- 理论题 900 道。
- 实操题 161 道。
- 解读总数 1061 条且尽量逐题唯一。
- 错题解释必须来自逐题解读数据。
- 修改后优先改生成源脚本，再构建发布文件。
- 发布后验证线上链接确实更新。

## 三点七、知识同步原则

可以同步到 GitHub 的是知识层，不是项目层：

- 可同步：`AGENTS.md`、全局控制台、SOP、复利日志、工作区提示词、hooks、可复用 skills、模板。
- 默认不同步：真实项目目录、题库数据、视频音频、大文件、`.env`、token、密码、客户/个人数据。
- 公司电脑需要复用时，优先克隆知识仓库，再按本机路径创建新的 workspace。
- 自建或深度个人化的轻量 skill 可以同步到仓库根目录 `skills/`；外部下载的完整开源仓库只保留学习结论或个人化后的精简 skill，不整包同步。

同步目标：

- 最终目标是让公司电脑能直接克隆这些全局 md 和规则文件，继承用户偏好、复盘历史、预览优先习惯、项目 SOP、hooks 和协作方式。
- 不把具体业务项目当成记忆同步；项目只在必要时单独迁移。

## 四、可复用提示词模板

### 商业化或自动化项目启动

```text
当项目带有商业化、自动化、增长、变现目标时，必须先定义：
1. 目标用户
2. 交付物
3. 变现路径
4. 验证指标
5. 最小可行版本
6. 风险边界
7. 停止条件
```

### 新项目创建

```text
请在 D:\Workspace 下创建一个新项目，项目名称是【项目名称】。
要求使用 Project_编号_项目名称 命名，自动递增编号，并创建标准项目目录结构。
所有后续相关文件都必须放入该项目文件夹。
```

### 阶段复盘

```text
请复盘当前项目阶段，整理完成内容、踩坑问题、可复用方法，并分别更新项目内复盘日志和全局复利踩坑日志。
```

### 工作区整理

```text
请检查 D:\Workspace 是否存在散落文件、命名不规范项目、缺失标准目录的项目，并给出整理建议。未经确认不要移动或删除文件。
```

### Claude Code 与 Codex 协作

```text
当 Claude Code、Codex 或其它模型协作时：
1. 执行方限制在当前项目文件夹或指定 worktree 中。
2. Review 方围绕 diff、测试结果和用户目标验收。
3. 自动化流程必须有停止条件、验证命令和人工验收环节。
4. 未经 review 的自动化结果不得直接合并、发布或覆盖重要文件。
```
