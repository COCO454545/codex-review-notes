# 00_全局控制台

> 本文档是 `D:\Workspace` 的全局工作台入口。Codex 在本 Workspace 执行任何项目型任务前，应优先遵守本文档、`AGENTS.md` 和 `03_新项目创建SOP.md`。

## 一、这个体系的好处

1. **所有内容有固定归属**：每个项目都有独立文件夹，需求、素材、代码、输出和复盘不会散落。
2. **Codex 接手更快**：下一次继续项目时，先读项目简报和日志就能恢复上下文。
3. **减少重复踩坑**：错误、弯路和可复用方法沉淀到全局日志，后续项目可以直接复用。
4. **方便备份和迁移**：项目文件夹天然就是一个完整包，复制、归档、交付都更简单。
5. **项目越多越清楚**：通过编号、项目索引、状态字段，可以快速知道哪些项目在做、暂停、完成或归档。
6. **避免临时文件污染电脑**：素材、截图、下载、导出、测试结果都有明确位置。

## 二、当前规则的不足与调整

| 原不足 | 调整方式 |
| --- | --- |
| 只有项目文件夹，没有全局项目索引 | 新增 `02_全局项目索引.md`，记录项目编号、状态、路径和下一步 |
| 所有任务都建项目可能太重 | 增加“轻量任务例外”：一次性问答、翻译、简单命令不强制建项目 |
| 缺少项目状态管理 | 增加状态：`规划中`、`进行中`、`暂停`、`已完成`、`已归档` |
| 缺少归档规则 | 完成项目需要整理 `05_outputs`、`06_tests`、`08_review`，再更新项目索引 |
| 缺少外部文件处理规则 | 外部素材应复制或移动进项目目录，并在引用资料中注明来源 |
| 踩坑日志容易变流水账 | 增加“记录门槛”：只有可复用、可避免、影响结果的问题才进入全局日志 |
| 全局复盘误建项目文件夹 | 明确 Codex 使用复盘、偏好沉淀、全局 SOP 更新、GitHub 同步规则不建新项目，直接更新根目录全局 md |

## 三、Workspace 总原则

1. `D:\Workspace` 是默认工作根目录。
2. 除非用户明确指定，否则项目相关内容不得创建到桌面、下载目录、临时目录、系统目录或其它磁盘位置。
3. 新项目必须创建独立项目文件夹，命名为 `Project_编号_项目名称`。
4. 根目录只存放全局文档、`AGENTS.md` 和项目文件夹。
5. 项目相关内容必须放入对应项目文件夹，包括需求、资料、素材、代码、脚本、图片、视频、导出结果、测试记录、日志和复盘。
6. 如果任务很小且不会产生项目文件，可以不创建项目；一旦产生多个文件或需要后续维护，就必须创建项目。
7. 如果任务是 Workspace 规则、Codex 使用体验复盘、全局偏好沉淀、AGENTS/SOP/hooks/skills 调整或 GitHub 同步规则，不创建普通项目文件夹，直接更新根目录全局文档。

## 六、项目命名规范

项目文件夹统一使用：

```text
Project_编号_项目名称
```

示例：

```text
Project_01_Codex_Agent
Project_02_d_CN_MVP
Project_03_Guard_Global
Project_04_工作流实验
```

编号规则：

1. 编号使用两位数字，例如 `01`、`02`、`03`。
2. 新项目取当前最大编号加一。
3. `Project_00_全局工作台` 保留为全局资料文件夹，不作为普通项目编号。
4. 项目名称要短、清晰、可搜索。
5. 项目名称可使用中文、英文、数字和下划线。
6. 不使用空格、斜杠、冒号、问号、星号等特殊符号。

## 七、项目内部目录结构

```text
Project_编号_项目名称
├─ 00_project_brief.md
├─ 01_requirements
├─ 02_references
├─ 03_assets
├─ 04_src
├─ 05_outputs
├─ 06_tests
├─ 07_logs
├─ 08_review
└─ 99_temp
```

目录用途：

- `00_project_brief.md`：项目目标、范围、当前状态、关键文件、下一步。
- `01_requirements`：需求、任务说明、验收标准、变更记录。
- `02_references`：参考资料、截图、网页、竞品、来源说明。
- `03_assets`：图片、音频、视频、字体、Logo、原始素材。
- `04_src`：代码、脚本、配置、工程文件。
- `05_outputs`：最终交付物、导出文件、发布版本。
- `06_tests`：测试记录、截图、验证结果。
- `07_logs`：执行过程、关键命令、报错记录。
- `08_review`：复盘、反馈、改进建议。
- `99_temp`：短期临时文件，完成后清理或归档。

## 八、Codex 执行规则

Codex 在本 Workspace 中工作时必须遵守：

1. 先判断任务是否属于已有项目。
2. 如果是已有项目，先阅读 `00_project_brief.md` 和最近日志。
3. 如果是新项目，先按 SOP 创建 `Project_编号_项目名称`。
4. 所有项目文件必须放入对应项目目录，不能散落在根目录。
5. 生成最终文件后，放入 `05_outputs`。
6. 过程验证和截图放入 `06_tests`。
7. 执行日志、错误记录、关键命令放入 `07_logs`。
8. 可复用踩坑写入 `01_全局复利踩坑日志.md`。
9. 每次创建、暂停、完成或归档项目时，更新 `02_全局项目索引.md`。
10. 不确定项目归属时，优先询问用户；如果用户已给出明确项目名，则直接执行。
11. 项目任务开始前，明确目标、假设、输出位置和验证方式。
12. 项目任务完成后，说明改动内容、文件路径和验证结果。
13. 优先使用最简单、最少改动、最容易验证的方案。
14. 不新增未被要求的功能、配置、抽象或复杂流程。
15. 不修改无关文件，不清理无关内容，不擅自移动用户文件。
16. Workspace hooks 是机械闸门：如果 hooks 提示路径越界、缺少 proof、缺少质检或输出未复盘，应先补齐证据再继续。
17. 视频项目在 full render 前必须先完成低成本 proof，包括参考学习、视频规格、开场/风格预览和男性口播 proof。
18. UI、App、系统交互和视频类任务，除单张图片生成外，默认先做 2-4 个卡片总览/风格预览方向，让用户选中后再进入完整实现或完整渲染。
19. 全局规则复盘只更新根目录全局 md，不创建 `Project_*` 文件夹；只有真实业务项目、工具、视频、应用、调研交付物才创建项目文件夹。
20. Workspace 知识同步 GitHub 时，只同步全局 md、hooks、模板和可复用 rules；默认排除 `Project_*`、大文件、题库、输出物和敏感资料。
21. 非简单任务先做能力发现：优先检查 Codex 已有 skills、插件、apps/connectors、MCP 工具、hooks、已有脚本和 `D:\ai_tool\skills`；不足时再查 GitHub、官方文档和成熟开源项目，并把高价值模式沉淀成个人 skill、脚本模板或全局规则。
22. 安全、低风险、可回滚的小任务可以直接执行，不反复打断用户确认；涉及删除、上传、真实账号、生产环境、第三方平台或不可逆操作时，必须先说明风险和影响范围并获得确认。
23. 修改完成后的汇报必须包含：改了什么、影响范围、验证结果；无法验证时说明原因和剩余风险。
24. 自建或深度个人化的轻量 skills 可镜像到根目录 `skills/` 并同步 GitHub；第三方完整仓库、训练数据、大媒体、缓存和依赖目录默认不同步。

## 九、安全与协作底线

1. 不得将密码、密钥、API key、token 等敏感信息写入代码或日志。
2. 不得擅自提交 `.env` 等本地配置文件。
3. 日志、截图、提交说明和复盘中不得输出隐私信息或敏感信息。
4. 涉及删除、批量移动、上传数据、生产环境、真实账号、自动发布等高风险操作前，必须先获得用户明确确认。
5. 修复问题后必须进行验证；无法验证时，要说明原因、已完成检查和剩余风险。
6. 合并、交付或采纳自动化结果前，必须查看关键 diff 或产出内容。
7. Claude Code、Codex 或其它模型协作时，应有明确执行边界、验证命令、停止条件和人工验收环节。

## 十、轻量任务例外

以下任务不强制创建项目：

1. 简单问答、解释概念、翻译一句话。
2. 查看当前时间、运行一次简单命令。
3. 不产生文件、不需要后续维护的一次性任务。

但如果轻量任务开始产生文件、素材、脚本、输出或复盘价值，应立即归入已有项目或创建新项目。
