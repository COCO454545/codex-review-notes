# 00_全局控制台

> 本文档是 `D:\Workspace` 的最高入口。这里定义了 Workspace 的沉淀边界、项目工作方式、文件归属规则，以及跨电脑同步时真正要长期保留的成果。

## 一、核心原则

1. **`D:\Workspace` 是唯一沉淀真源。**
2. 真正长期保留、可跨电脑同步的成果，只有两类：
   - Workspace 根目录的全局 Markdown 文档
   - `D:\Workspace\skills\` 下的自建 skills
3. `D:\Workspace\Project_*` 下的普通项目内容，属于项目层，不属于全局沉淀层，默认不同步到 GitHub。
4. 任何新规则、长期偏好、复盘方法、可复用流程，都优先沉淀到根目录全局 md 或 `D:\Workspace\skills\`。
5. 结构性问题如果不明确，先确认，不擅自扩展目录体系。

## 二、Workspace 的作用

`D:\Workspace` 同时承担两种职能：

### 1. 日常工作根目录
这里包含真实项目目录，例如：
- `Project_01_*`
- `Project_02_*`
- `Project_03_*`
- ...

这些目录用于承载具体项目的需求、素材、代码、输出、测试和复盘。

### 2. 长期沉淀根目录
这里还包含一层不会随着项目结束而失效的长期成果：
- 全局规则
- 长期提示词
- 踩坑与方法论
- 项目创建 SOP
- 跨电脑同步规范
- 自建 skills

后者才是需要长期保留并同步到远程仓库的“知识层”。


- 根目录 md + `skills/` = 长期沉淀层
- `Project_*` = 项目层
- GitHub 只同步沉淀层，不同步普通项目层

## 五、项目命名规范

普通项目目录统一使用：

```text
Project_编号_项目名称
```

当前统一规则：
1. 编号使用两位数字，如 `01`、`02`、`03`。
2. 新项目编号取当前最大编号加一。
3. 项目名称要短、清晰、可搜索。
4. 可使用中文、英文、数字、下划线。
5. 不使用空格、斜杠、冒号、问号、星号等特殊符号。

## 六、项目层与沉淀层的边界

### 属于项目层（默认不同步）
- `Project_*` 下的代码、素材、数据、图片、视频、导出物
- 题库原始数据、客户资料、账号信息
- node_modules、dist、build、缓存、临时文件
- 任何具体业务项目的过程性资产

### 属于沉淀层（可以同步）
- Workspace 根目录全局 md
- `D:\Workspace\skills\` 下的 skills
- 已经提炼完成的方法论、模板、规则

## 七、Codex 执行规则

Codex 在 `D:\Workspace` 工作时必须遵守：

1. 先判断当前任务属于**项目层**还是**沉淀层**。
2. 如果属于项目层，先确认是否已有对应 `Project_*`，没有则按 SOP 创建。
3. 如果属于沉淀层（规则更新、偏好沉淀、同步规范、skills 调整），直接更新根目录全局 md 或 `skills/`，不创建新项目。
4. 所有项目相关文件必须放入对应项目目录，不得散落在根目录。
5. 根目录只放全局文档和 `skills/` 这类长期资产。
6. 能复用的经验要沉淀成规则或 skill，不停留在一次性对话里。
7. 优先使用最简单、最少改动、最容易验证的方案。
8. 不新增无关结构，不擅自发明新目录体系。

## 八、开始前检查清单

- [ ] 当前任务是项目层还是沉淀层？
- [ ] 是否需要创建或进入某个 `Project_*`？
- [ ] 如果是沉淀层任务，是否应该直接更新根目录 md 或 `skills/`？
- [ ] 输出位置是否明确？
- [ ] 是否知道如何验证结果？

## 九、完成前检查清单

- [ ] 项目文件是否都放在了正确项目目录？
- [ ] 全局规则是否只更新到了根目录 md？
- [ ] 新增 skill 是否放在了 `D:\Workspace\skills\skill-name\SKILL.md`？
- [ ] 可复用经验是否已沉淀？
- [ ] 是否没有把项目层内容误放进沉淀层？

## 十、核心思维

AI 时代更高效的做法不是“每次从零开始”，而是：

> **让工作中的规则、偏好、方法和技能，持续沉淀成一套可跨电脑复用的长期成果。**
