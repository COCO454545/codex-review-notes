# 调研分析框架

遵循 `00_全局控制台.md` 规范，提供标准化的技术选型和项目调研方法论。

## 何时使用

当用户说：
- "调研一下这个项目..."
- "评估一下这个技术方案..."
- "对比一下这些项目..."
- "分析一下做 XX 的可行性..."
- "帮我看看 GitHub 上的这个项目..."

不要用于：
- 简单的代码编写任务
- 不需要深度分析的快速验证

## 核心分析维度

### 1. 需求拆解矩阵

第一步永远是先明确用户的真实需求：

| 模块 | 说明 | 是否必须 | 优先级 |
|------|------|---------|--------|
| 核心功能1 | ... | 必须 | P0 |
| 核心功能2 | ... | 必须 | P0 |
| 增强功能1 | ... | 建议 | P1 |
| 增强功能2 | ... | 建议 | P1 |
| 未来扩展 | ... | 可选 | P2 |

### 2. 项目总体判断

```
没有一个项目可以 100% 开箱即用。比较合理的路线是：
1. 用一个现成的系统作为第一阶段底座
2. 二开关键业务逻辑
3. 二开增强功能
4. 二开分析与运营面板
```

### 3. 候选项目优先级排序

| 优先级 | 项目 | 推荐原因 | 适配度评分 |
|--------|------|---------|-----------|
| 1 | org/repo | ... | 高/中/低 |
| 2 | org/repo | ... | 高/中/低 |

### 4. 项目逐个分析模板

对每个候选项目按以下维度分析：

```markdown
### X.X 组织名/项目名

项目地址：https://github.com/组织名/项目名

**定位**：一句话说明项目定位

**能解决**：
- ✅ 能力1
- ✅ 能力2

**不能完全解决**：
- ⚠️ 能力3（需要实测）
- ⚠️ 能力4（需要二开）

**需要二次开发**：
- 🔧 需要新增 XX 功能
- 🔧 需要改造 YY 模块

**适配你的需求**：

| 能力 | 适配度 |
|------|--------|
| 多账号管理 | 高/中/低 |
| 核心功能1 | 高/中/低 |
| 核心功能2 | 高/中/低，需要二开 |

**结论**：最适合作为第一阶段试跑底座 / 适合做 XX 参考 / 不建议作为主系统
```

### 5. 关键痛点深度分析

针对用户最关心的 3-5 个痛点，分别分析：
- 现成项目能解决到什么程度
- 仍需关注什么风险
- 建议如何二开
- 推荐的规则或策略

### 6. 技术实现方案建议

对核心但不现成的功能，给出最小实现方案：

```markdown
## X. XX 功能方案

### 基础流程
1. ...
2. ...
3. ...

### 关键状态机
| 状态 | 含义 |
|------|------|
| state1 | ... |
| state2 | ... |

### 数据表建议
| 字段 | 说明 |
|------|------|
| field1 | ... |

### 推荐规则
| 场景 | 推荐规则 |
|------|---------|
| 场景1 | ... |

### 技术实现
最小实现：...
更安全/更完整实现：...
```

### 7. 推荐实施路线（四阶段）

```markdown
## 阶段 1：最小可用系统

目标：先跑通 1-2 个核心场景
验证标准：
- [ ] 能完成核心流程 A
- [ ] 能完成核心流程 B
- [ ] 关键数据能正确记录

## 阶段 2：按业务二开

优先二开：
1. 功能 A
2. 功能 B
3. 功能 C

## 阶段 3：稳定性和规模化

当数据/用户变多后再做：
- 数据库优化
- 引入缓存/任务队列
- 健康监控
- 限流与错峰

## 阶段 4：分析与运营

二开分析面板：
- 按 XX 看表现
- 按 YY 看转化率
- 输出操作建议
```

### 8. 最终推荐总结

```markdown
## 最终推荐

如果只选一个项目先试：
> 优先试 xxx/xxx

如果想对比 XX 能力：
> 同时试 xxx/xxx

如果更偏 XX 技术栈：
> 试 xxx/xxx

如果以后要完全自研：
> 用 xxx/xxx 做底层
```

### 9. 第一版系统边界（非常重要）

明确列出：
✅ **第一版只做这 5 件事**：
1. ...
2. ...
3. ...
4. ...
5. ...

❌ **暂时不要优先做**：
- ...
- ...
- ...

## 输出格式要求

调研报告必须包含：
1. ✅ 需求拆解矩阵
2. ✅ 总体判断与候选排序
3. ✅ 每个候选项目的深度分析
4. ✅ 关键痛点与二开建议
5. ✅ 核心功能的技术实现方案
6. ✅ 四阶段实施路线
7. ✅ 最终推荐与系统边界
8. ✅ 参考链接清单

## 注意事项

- 永远不要只列优点，必须同时列风险和不足
- 对"需要二开"的部分要具体说明改什么
- 第一版系统边界要保守，能自动的自动，风险高的提醒人工
- 对商业使用许可、平台规则风险要明确标注
- 如果项目有"仅供学习研究"声明，必须提醒用户注意
