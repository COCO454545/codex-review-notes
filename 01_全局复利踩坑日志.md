# 01_全局复利踩坑日志

> 本文档用于记录 Workspace 中反复出现的问题、踩过的坑、走过的弯路，以及未来可以复用的方法。目标不是记流水账，而是把错误转化为下一次的速度。

## 一、记录门槛

只有满足以下任一条件，才写入全局日志：

1. 这个错误未来可能再次发生。
2. 这个坑影响了结果、效率、质量或文件归档。
3. 这次形成了可复用方法、命令、模板或检查清单。
4. 这个经验对多个项目都有价值。

只属于单个项目、没有复用价值的细节，放到项目内 `07_logs` 即可。

## 二、使用规则

1. 每条记录必须写清楚“下次如何避免”。
2. 不只记录失败，也记录有效方法。
3. 项目内细节日志放在对应项目的 `07_logs`。
4. 稳定可复用的方法沉淀到“复利方法库”。
5. 新项目开始前，Codex 应快速检查本文档是否有相关经验。

## 九、复利方法库

### 方法 001：所有项目先建文件夹，再开始工作

适用场景：

- 新项目、新工具、新视频、新脚本、新调研、新实验。

执行步骤：

1. 在 Workspace 根目录创建 `Project_编号_项目名称`。
2. 创建标准目录。
3. 新增 `00_project_brief.md`。
4. 更新 `02_全局项目索引.md`。
5. 后续所有文件都放入该项目文件夹。

检查标准：

- 根目录没有项目散落文件。
- 项目文件夹内有简报、输出、日志和复盘位置。
- 全局项目索引可以找到这个项目。

复用价值：

- 避免文件散落。
- 方便复盘、迁移、备份和继续开发。
- 让 Codex 每次接手都有上下文。

### 方法 002：先判断任务重量，再决定是否建项目

适用场景：

- 用户临时提出一个小任务，不确定是否需要创建项目。

执行步骤：

1. 如果只是问答、翻译、一次性命令，不建项目。
2. 如果会产生多个文件、素材、脚本、导出物或需要后续维护，创建项目。
3. 如果任务从小变大，立即补建项目并把相关文件归入项目目录。

检查标准：

- 小任务不被过度流程化。
- 大任务不丢上下文和文件。

复用价值：

- 保持系统干净，同时避免流程太重。

### 方法 003：用 Workspace hooks 拦截机械性重复错误

适用场景：

- 文件容易散落到 Workspace 根目录、桌面、下载目录或临时目录。
- 视频项目容易跳过 proof 直接 full render，浪费额度和时间。
- Codex 容易在缺少质检、复盘或预览证据时过早说“完成”。

执行步骤：

1. 在 `D:\Workspace\.codex\hooks.json` 配置项目级 hooks。
2. `UserPromptSubmit` 负责提醒项目 SOP 和视频 SOP。
3. `PreToolUse` 负责拦截根目录污染、危险删除和未过 proof gate 的 full render。
4. `Stop` 负责在交付前检查项目输出、质检记录、预览帧和复盘证据。
5. hooks 更新后，在 Codex 的 hooks 管理界面 review/trust 新 hook。

检查标准：

- 项目文件不会直接写到 Workspace 根目录。
- full render 前已经有 `video-spec.md`、`reference-learning.md`、开场/风格 proof 和男性口播 proof。
- 有输出文件时，必须有 `06_tests` 或 `08_review` 里的验证/复盘证据。

复用价值：

- 把“靠记忆执行”的规则变成“自动提醒/自动拦截”的机制。
- 降低重复踩坑、文件混乱和高成本返工。
- 让 AGENTS.md、SOP、skill 和 hooks 形成分层：原则写 AGENTS，流程写 skill，机械错误由 hooks 拦截。

### 方法 004：视觉和视频任务先做预览 proof，再做最终交付

适用场景：

- 网站、App、小程序、管理后台、系统交互、视频、口播、海报等风格敏感任务。
- 用户给了参考图、竞品、链接或明确审美偏好。
- 返工成本高、full render 成本高、部署成本高的任务。
- 单张图片生成任务默认不强制执行本方法，除非用户明确要求多风格候选。

执行步骤：

1. 先读用户参考图、PRD、历史复盘和已有项目。
2. 再查 GitHub、官方文档、成熟项目或相关 skill。
3. UI、App、系统交互、网站类任务先做 2-4 个类似“卡片总览”的预览板，展示页面组合、信息架构、视觉关键词、组件风格和交互气质。
4. 视频任务先做 2-4 个预览卡片或开场风格帧，再做分镜和 10-15 秒口播 proof。
5. 用户确认方向后，再进入完整开发、完整渲染或完整部署。
6. 如果预览方向不对，换方向，不在错误方向上继续消耗额度。

检查标准：

- 有 `reference-learning.md` 或等价参考分析。
- 有低成本预览或 proof。
- 用户已确认方向，或任务本身不依赖审美选择。
- full render / 完整开发 / 线上部署前，已经过预览闸门。

复用价值：

- 避免“做完才发现不是用户想要的”。
- 降低 token、额度、时间浪费。
- 把审美选择提前到低成本阶段。

### 方法 005：知识层同步 GitHub，项目层默认不同步

适用场景：

- 用户希望家里电脑和公司电脑共享 Codex 习惯、复盘、SOP、hooks、skills。
- 工作区中有大量项目文件、私密资料、大文件，不适合整体上传。

执行步骤：

1. 建一个私有 GitHub 仓库作为 Codex 个人知识层。
2. 同步 `AGENTS.md`、全局控制台、SOP、复利日志、工作区提示词、hooks、可复用 skills 和模板。
3. 默认排除普通 `Project_*`、题库数据、视频音频、PDF、`.env`、token、密码、客户/个人数据；`Project_00_全局工作台` 只允许放无敏感模板和共享材料。
4. 公司电脑克隆知识仓库后，再复制或软链接到新的 workspace。
5. 项目文件仍按本机项目目录单独管理。
6. 自建或深度个人化的轻量 skill 可以放入仓库根目录 `skills/`；第三方完整仓库、依赖目录、训练数据、大媒体和缓存默认不同步。

检查标准：

- 仓库中没有密钥、密码、真实用户数据。
- 仓库中没有大体积输出物。
- 公司电脑克隆后能恢复 Codex 的规则、偏好和 hooks。

复用价值：

- 减少多设备重复调教。
- 保留长期工作习惯和复盘历史。
- 避免把真实项目资料误传到 GitHub。

### 方法 006：全局复盘不创建项目文件夹

适用场景：

- 用户要求复盘 Codex 使用体验、沉淀偏好、更新 AGENTS/SOP/hooks/skills。
- 用户要求整理 Workspace 全局规则或 GitHub 同步规则。
- 输出物是全局 md 规则，而不是业务项目、工具、视频、应用或调研交付物。

执行步骤：

1. 直接更新根目录全局 md：`00_全局控制台.md`、`01_全局复利踩坑日志.md`、`03_新项目创建SOP.md`、`06_工作区提示词.md`、`AGENTS.md` 等。
2. 不创建新的 `Project_*` 文件夹。
3. 如需保存截图参考，只在确有长期价值时放入 `Project_00_全局工作台` 或转写成文字规则。
4. 更新 GitHub 同步 SOP 时，直接改 `07_GitHub同步SOP.md`。

检查标准：

- 根目录全局 md 已更新。
- `02_全局项目索引.md` 没有新增“复盘类项目”。
- 没有产生多余的普通项目文件夹。

复用价值：

- 避免为了全局规则复盘而制造无意义项目。
- 让公司电脑同步时只拿到真正有用的知识层。

### 方法 007：先查 Codex 能力和开源 Skills，再做个人化沉淀

适用场景：

- 用户要做视频流水线、自动化、App、系统、插件、复杂脚本、调研分析或可复用工作流。
- 任务可能已经有成熟 skill、插件、connector、MCP 工具、脚本或 GitHub 开源项目可以借鉴。
- 用户明确希望 Codex 不要闭门造车，而是吸收主流优秀方案后再进化成个人流程。

执行步骤：

1. 先查本地 Codex 能力：当前会话可用 skills、插件、apps/connectors、MCP 工具、workspace hooks、已有脚本、`D:\ai_tool\skills` 下的个人 skills。
2. 如果找到可用能力，先按该 skill/plugin/script 的说明执行，并说明采用原因。
3. 如果本地能力不足，再查一手资料：GitHub、官方文档、成熟开源项目、demo、issues、releases、主流社区实践。
4. 评估开源方案时，看使用人数、star/增长、最近维护、示例完整度、license、依赖复杂度、是否适合当前项目。
5. 不直接照搬。把适合的模式提炼成个人项目 skill、脚本模板、SOP、hooks 规则或全局提示词。
6. 如果沉淀的是 Workspace 级规则，直接更新根目录全局 md；如果沉淀的是某个真实项目能力，放进对应项目或 `D:\ai_tool\skills`。

检查标准：

- 开始实现前已经说明或记录“使用了哪个现成能力/参考了哪些一手资料/为什么不用现成方案”。
- 高价值发现已经沉淀，不停留在一次性聊天结论。
- 没有为了追新工具而引入不必要复杂度。

复用价值：

- 减少重复造轮子。
- 让 Codex 更像会选工具的工作台，而不是每次从零开始。
- 把外部优秀实践转化成用户自己的长期技能资产。

### 方法 008：按风险分级执行，低风险直接做，高风险先确认

适用场景：

- 用户希望 Codex 主动完成任务，而不是只给手动清单。
- 任务包含文件修改、命令执行、GitHub 同步、部署、删除、上传、真实账号或生产环境操作。
- 需要在“少打扰用户”和“安全边界”之间取得平衡。

执行步骤：

1. 先判断风险等级：是否可回滚、是否只影响当前 Workspace、是否涉及外部平台、是否会删除或覆盖数据、是否接触敏感信息。
2. 低风险、可回滚、小范围任务直接执行，并在完成后说明改动、影响范围和验证结果。
3. 中等风险但可回滚的任务，先简短说明假设，再按最小可用改动推进。
4. 高风险任务先确认，包括删除、批量移动、上传数据、真实账号、生产环境、第三方平台、自动发布、不可逆写入。
5. 命令失败、测试失败或检查未通过时，如实记录具体结果，继续做下一步合理诊断。

检查标准：

- 没有把低风险任务变成反复确认。
- 没有在高风险任务上擅自执行。
- 最终汇报包含“做了什么、影响范围、验证了什么、剩余风险”。

复用价值：

- 降低沟通疲劳。
- 保留安全边界。
- 让 Codex 的执行方式更贴近用户偏好：能直接做就直接做，该刹车时必须刹车。

## 2026-06-28｜项目：全局工作台｜主题：参考优先、预览优先和知识同步

### 问题

近期 Codex 在人工智能培训师项目、视频生产和界面设计中出现过效果不稳定、直接进入最终实现、参考学习不足、用户需要反复调教的问题。
同时，本次复盘曾被误建成普通项目文件夹，和用户“全局复盘只沉淀到 workspace 根目录 md”的真实意图不一致。

### 原因

缺少四个前置闸门：Codex 能力发现、一手资料学习、低成本预览 proof、项目不变量确认。家里电脑沉淀的 AGENTS、复盘、hooks、skills 也没有形成可迁移的知识层，导致公司电脑可能重复调教。

### 解决方案

新增全局规则：界面、App、系统交互和视频任务先做 2-4 个预览方向，用户确认后再完整实现；复杂项目先查 Codex 已有 skills、插件、apps/connectors、MCP 工具、hooks、脚本和 `D:\ai_tool\skills`，再看 GitHub、官方文档和主流项目；高价值方案沉淀成个人 skill、脚本模板或全局规则；知识层可同步 GitHub，真实项目层默认不同步；全局复盘不创建普通项目文件夹。

### 下次避免

不要上来就做最终稿。先查现成能力、先参考、先预览、先 proof、先锁不变量，再开发、渲染或部署。跨电脑同步只同步知识层，不同步真实项目和敏感数据。
当用户是在复盘 Codex 使用方式、更新 workspace 规则、整理偏好时，直接更新全局 md，不新建 `Project_*`。

## 2026-06-29｜项目：Project_03_fuye 闲鱼多账号托管调研｜主题：第一版不要追求大而全

### 问题

在调研闲鱼自动化项目时，一开始想把所有功能都塞进第一版——多账号矩阵、销售分析、选品分析、卡密仓库、下载站、飞书通知、微信 Bot，觉得"反正以后都要做，不如现在一起做了"。

### 原因

对"最小可用"的理解不够深刻，把"未来阶段的功能"和"第一阶段必须的功能"混在了一起。没有明确的边界意识。

### 造成的影响

- 第一版复杂度飙升，可能还没跑起来就放弃了
- 无法快速验证核心假设（这个副业模式到底行不行）
- 技术债和维护成本一开始就很高

### 解决方案

明确第一版只做 5 件事：
1. 多账号绑定和状态监控
2. 商品级自动回复
3. 虚拟商品自动发货
4. 实物商品订单推送到飞书/微信 Bot
5. 一次性取件码系统

其他所有功能全部延后到阶段 3、4 再做。

### 下次避免

- 任何项目第一版启动前，先写清楚"只做哪 5 件事"
- 再写清楚"暂时不做哪些事"，这比做什么更重要
- 每个阶段都要有明确的停止条件，完成就进入下一个阶段，不要边做边加

### 可复用方法

- 5件事原则：第一版功能不要超过 5 个核心点
- 写两份清单：做什么 + 不做什么
- 四阶段实施路线：最小可用 → 业务二开 → 稳定规模化 → 数据运营

- 项目调研沉淀在 `Project_03_fuye/`

## 十一、优质做法记录

### 做法 001：7维度技术选型评估法

日期：2026-06-29

关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
对开源项目进行系统化选型评估，按 7 个维度完整覆盖：
1. 需求拆解 → 明确核心/非核心需求，做成矩阵表格
2. 总体判断 → 先给结论，没有完美项目，只有阶段最优方案
3. 逐个项目分析 → 每个项目独立分析：定位、能解决什么、不能解决什么、需要二开什么、适配度评分
4. 关键痛点深度分析 → 针对每个核心需求，横向对比所有候选项目的覆盖程度
5. 核心功能方案建议 → 对需要二开的功能，给出具体实现思路（状态机、数据表、规则设计等）
6. 四阶段实施路线 → 最小可用 → 业务二开 → 稳定规模化 → 数据运营分析
7. 明确第一版系统边界 → 只做 5 件事，剩下的留到后面阶段

为什么效果好：
- 避免了"这个项目功能多所以选它"的盲目选型，而是按需求匹配度判断
- 每个维度都有可落地的产出，不是空泛的评价
- 边界定义避免了第一版就陷入"大而全陷阱"

适用场景：
- 开源项目技术选型
- 复杂系统第一版方案设计
- 需求不明确时的方案调研
- 副业自动化项目方案评估

复用步骤：
1. 先列需求矩阵（模块、说明、是否必须）
2. 候选项目优先级排序，给出推荐原因
3. 每个项目按"定位→能解决→不能解决→需要二开→适配度"五要素分析
4. 按核心痛点横向对比，发现所有项目都没覆盖的就是你需要重点二开的地方
5. 给出分阶段实施路线图
6. 最后明确第一版的边界：只做什么，不做什么

注意事项：
- 一定要有"不能解决"和"需要二开"的部分，避坑比夸优点更重要
- 第一版边界越清晰越好，宁可少做，不要错做
- 商业使用许可、平台规则、风控风险等非技术因素也要考虑

### 做法 002：副业自动化边界设计哲学

日期：2026-06-29

关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
副业自动化系统按"风险分层"设计边界：
- 低风险、高重复 → 100% 自动（虚拟商品发货、关键词回复）
- 中风险、偶尔发生 → 提醒人工（退款、投诉、异常订单）
- 高风险、涉及钱/规则 → 必须人工（实物发货、客诉处理）

核心原则：能自动的自动，风险高的坚决只提醒人工。

7 条风险红线：
1. 退款、投诉、假货相关消息禁止 AI 自行承诺
2. 高价值商品延时发货或人工复核
3. 不要自动处理退货退款
4. 不要自动删除任何数据
5. 同一订单疑似重复发货时必须拦截
6. 账号掉线、Token 失效时只提醒，不要自动尝试无限重连
7. 不要把平台 Cookie 存在第三方服务器

为什么效果好：
- 把 AI/自动化定位成"帮手"而不是"替代者"
- 减少人工 80% 的重复工作，但保留人对 20% 风险的控制权
- 出问题时有人在，系统不背锅，平台不风控

适用场景：
- 任何涉及电商、支付、平台规则的自动化系统
- 副业自动化工具设计
- 需要对接第三方平台 API 的系统

复用步骤：
1. 列全所有业务场景
2. 按风险程度（低/中/高）给每个场景打分
3. 低风险 → 全自动
4. 中风险 → 系统推送通知 + 按钮，人点一下确认
5. 高风险 → 只给人发消息，系统什么都不做

注意事项：
- 不要为了追求"全自动"而把风险高的场景也自动化
- 用户问"能不能全自动退款"时，宁可说"我给你推提醒，你点一下"，也不要说"能"
- 边界是给用户安全感，也是给系统擦屁股的空间

是否需要更新 AGENTS.md 或工作区提示词：建议加入 AGENTS.md 的安全要求部分

### 做法 003：一次性取件码状态机设计

日期：2026-06-29

关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
资源资料类虚拟商品，不直接发长期网盘链接，而是发"一次性取件码"，用 6 状态状态机控制全流程：

```
unused（未使用）→ locked（已分配给订单）→ used（已提取）→ expired（过期）
                                                        ↓
                                                  refunded（退款作废）
                                                        ↓
                                                  disabled（人工禁用）
```

10 字段数据表标准设计：
- code: 一次性取件码（随机字符串）
- product_id: 商品 ID
- account_id: 账号 ID
- order_id: 订单 ID
- buyer_id: 买家标识
- resource_id: 资源 ID
- status: 状态（6 种枚举）
- expire_at: 过期时间
- used_at: 首次提取时间
- max_downloads: 最大下载次数

按商品类型匹配规则：
- 低价资料 → 首次提取后失效，允许 1 小时内重复打开
- 高价资料 → 首次提取后立即失效，只展示一次
- 软件/LUT 包 → 允许 24 小时内最多下载 2 次
- 电影票/卡券 → 展示一次后立即失效

为什么效果好：
- 解决了资源链接长期外传的痛点
- 每个取件码都和订单、买家绑定，可追溯
- 退款后可以立即作废对应取件码
- 通过限制下载次数，平衡用户体验和防扩散

适用场景：
- 任何虚拟资料商品发货系统
- 付费下载站、知识付费平台
- 需要防分享的数字内容分发

复用步骤：
1. 独立一个取件码服务，不和主系统耦合
2. 设计上述 6 状态 + 10 字段数据表
3. 发货时调用取件码 API，返回链接给买家
4. 提取时校验状态，通过后才展示真实资源
5. 退款回调把对应取件码改成 refunded

注意事项：
- 真实资源建议放对象存储（OSS/COS/R2），生成签名 URL，不要暴露永久地址
- 补发也要走相同流程，人工生成新码，不要直接发原始链接
- 要考虑用户可能没看到、浏览器关闭等情况，给合理的重试窗口

是否需要更新 AGENTS.md 或工作区提示词：建议作为通用设计模式沉淀

### 做法 004：四阶段系统分层实施路线

日期：2026-06-29

关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
任何复杂系统都按四阶段渐进实施，不追求一步到位：

阶段 1：最小可用系统
- 目标：先跑通最核心的 1-2 个场景
- 验证标准：只测核心流程，其他功能都可以后期加
- 只选最成熟的底座项目，不要一开始就自研

阶段 2：按业务二开
- 目标：把你的业务逻辑叠加上去
- 优先二开：业务类型区分、订单路由规则、通知模板、核心增强功能

阶段 3：稳定性和规模化
- 目标：当数据/用户变多后，系统还能跑得动
- 做什么：数据库优化、引入缓存和任务队列、健康监控面板、限流错峰

阶段 4：分析和运营
- 目标：用数据指导业务，而不是纯靠感觉
- 做什么：销售分析、品类表现、选品分析、外部市场数据对接

为什么效果好：
- 每个阶段都有明确的停止条件，不会无限蔓延
- 阶段 1 就能看到效果，有正反馈，不会死在半路上
- 后面阶段是可选的，根据业务发展决定要不要做

适用场景：
- 任何从零开始的软件项目
- 副业项目从 0 到 1
- 开源项目二开定制

复用步骤：
1. 先定义阶段 1 的"最小可用"是什么，写清楚"只做哪 5 件事"
2. 阶段 1 跑通了，再列阶段 2 的二开清单，按优先级排序
3. 每进入下一个阶段前，先确认上一个阶段是稳定的
4. 永远不要跨阶段（比如阶段 1 还没跑通就开始做阶段 4 的数据分析）

注意事项：
- 最容易犯的错：把阶段 4 的功能拿到阶段 1 来做
- "以后可能需要"是危险信号，大概率你现在不需要
- 每个阶段的边界要写下来，不要边做边加功能
