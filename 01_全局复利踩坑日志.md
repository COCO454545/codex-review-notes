# 01_全局复利踩坑日志

> 本文档用于记录 Workspace 中反复出现的问题、踩过的坑、走过的弯路，以及未来可以复用的方法。目标不是记流水账，而是把错误转化为下一次的速度。

## 一、记录门槛

只有满足以下任一条件，才写入全局日志：

1. 这个错误未来可能再次发生。
2. 这个坑影响了结果、效率、质量或文件归档。
3. 这次形成了可复用方法、命令、模板或检查清单。
4. 这个经验对多个项目都有价值。

只属于单个项目、没有复用价值的细节，优先写入对应项目内的复盘与日志目录。

## 二、使用规则

1. 每条记录必须写清楚“下次如何避免”。
2. 不只记录失败，也记录有效方法。
3. 项目内细节先写项目日志，再把跨项目经验同步到这里。
4. 稳定可复用的方法沉淀到“复利方法库”。
5. 新项目开始前，优先快速检查本文档是否已有相关经验。

## 八、复利方法库

### 方法 001：所有项目先建文件夹，再开始工作

适用场景：
- 新项目、新工具、新视频、新脚本、新调研、新实验。

执行步骤：
1. 在 Workspace 根目录创建 `Project_编号_项目名称`。
2. 创建标准目录结构。
3. 新增 `00_project_brief.md`。
4. 更新 `02_全局项目索引.md`。
5. 后续所有项目文件都放入该项目目录。

检查标准：
- 根目录没有项目散落文件。
- 项目文件夹内有简报、输出、日志和复盘位置。
- 全局项目索引可以找到这个项目。

复用价值：
- 避免文件散落。
- 方便复盘、迁移、备份和继续开发。
- 让 AI 接手时始终有上下文。

### 方法 002：先判断任务重量，再决定是否建项目

适用场景：
- 用户临时提出一个小任务，不确定是否需要创建项目。

执行步骤：
1. 如果只是问答、翻译、一次性命令，不建项目。
2. 如果会产生多个文件、素材、脚本、导出物或需要后续维护，创建项目。
3. 如果任务从小变大，立即补建项目并把相关文件归入项目目录。

检查标准：
- 小任务不被过度流程化。
- 大任务不丢上下文和文件。

复用价值：
- 保持系统干净，同时避免流程太重。

### 方法 003：先查现有能力和一手资料，再做个人化沉淀

适用场景：
- 要做视频流水线、自动化、App、系统、复杂脚本、调研分析或可复用工作流。

执行步骤：
1. 先查本地已有能力：`D:\Workspace\skills\`、已有脚本、现成模板、已知工具。
2. 如果已有能力能覆盖需求，优先复用，不闭门造车。
3. 如果能力不足，再查 GitHub、官方文档、成熟开源项目、示例、issues、releases。
4. 找到高价值模式后，不直接照搬，提炼成自己的规则、模板或 skill。

检查标准：
- 开始实现前已经判断过“有没有现成能力可复用”。
- 最终沉淀的是自己的长期资产，而不是临时抄来的方案。

复用价值：
- 减少重复造轮子。
- 让沉淀越来越像自己的工作台，而不是一次性聊天记录。

### 方法 004：视觉和视频任务先做低成本预览 proof

适用场景：
- 网站、App、小程序、系统交互、视频、海报、口播等风格敏感任务。

执行步骤：
1. 先读用户给的参考图、链接、PRD、历史复盘。
2. 再查本地和外部已有能力。
3. 先输出 2-4 个低成本预览方向，而不是直接做最终成品。
4. 用户确认方向后，再进入完整实现或完整渲染。
5. 如果方向不对，先换方向，不在错误方向上继续消耗额度。

检查标准：
- 有参考学习或预览证明。
- 最终稿前已经过方向确认。

复用价值：
- 避免“做完才发现不是用户想要的”。
- 把审美判断提前到低成本阶段。

### 方法 005：Workspace 是真源，GitHub 只同步沉淀层

适用场景：
- 家里电脑和公司电脑共享长期沉淀时。

执行步骤：
1. 以 `D:\Workspace` 为唯一沉淀真源。
2. 只同步 Workspace 根目录全局 md 和 `D:\Workspace\skills\`。
3. 不同步普通 `Project_*`、大文件、密钥、真实数据。
4. 公司电脑克隆后，也以本机 `D:\Workspace` 作为使用入口，而不是直接在镜像目录里工作。

检查标准：
- 远程仓库里只有沉淀层，没有项目层污染。
- 两台电脑最终都使用同一套沉淀成果。

复用价值：
- 避免各自为战。
- 换电脑时能快速恢复同一套工作台。

### 方法 006：全局复盘不创建项目文件夹

适用场景：
- 复盘 Codex 使用体验、沉淀偏好、更新 AGENTS/SOP/skills/同步规则。

执行步骤：
1. 直接更新根目录全局 md 或 `D:\Workspace\skills\`。
2. 不为全局规则复盘新建 `Project_*`。
3. 如果只是某个项目的局部经验，优先留在项目内；跨项目后再同步到全局。

检查标准：
- 没有为了更新规则而制造无意义项目目录。
- 根目录全局 md 承担了真正的长期沉淀职责。

复用价值：
- 让沉淀层保持纯净。
- 避免“为了复盘再建一个项目”的反模式。

### 方法 007：一项技能一个文件夹，一个入口文件

适用场景：
- 新建、维护、同步任何自定义 skill。

执行步骤：
1. 统一放在 `D:\Workspace\skills\`。
2. 每个 skill 使用独立目录。
3. 入口文件固定为 `SKILL.md`。
4. 需要扩展内容时，再增加 `references/`、`assets/`、`scripts/`、`agents/`。

检查标准：
- 不再出现 `skills\xxx.md` 这种平铺式错误结构。
- 每个 skill 都能独立迁移、同步和维护。

复用价值：
- 结构清晰。
- 更适合跨电脑同步和后续扩展。

### 方法 008：按风险分级执行，低风险直接做，高风险先确认

适用场景：
- 任务包含文件修改、命令执行、同步、部署、删除、上传、真实账号或生产环境操作。

执行步骤：
1. 先判断风险等级：是否可回滚、是否只影响当前 Workspace、是否涉及外部平台、是否会删除或覆盖数据、是否接触敏感信息。
2. 低风险、可回滚、小范围任务直接执行，并在完成后说明改动、影响范围和验证结果。
3. 中等风险但可回滚的任务，先说明假设，再按最小可用改动推进。
4. 高风险任务先确认，包括删除、批量移动、上传数据、真实账号、生产环境、第三方平台、自动发布、不可逆写入。

检查标准：
- 没有把低风险任务变成反复确认。
- 没有在高风险任务上擅自执行。

复用价值：
- 降低沟通疲劳。
- 保留安全边界。

## 九、典型踩坑记录

### 2026-06-26｜项目：全局工作台｜主题：视频生产失败后增加质量闸门意识

### 问题
视频项目出现过口播假、参考学习不深、过早进入完整渲染、交付前缺少风格 proof 的问题，同时 Workspace 也长期存在文件容易重复和散落的风险。

### 原因
规则、skill 和习惯虽然存在，但没有被前置成稳定的执行闸门，任务复杂时容易跳过低成本验证步骤。

### 解决方案
把这些经验沉淀为统一规则：
- 视频与视觉任务先做低成本 proof。
- 根目录只保留沉淀层，项目文件不得散落。
- 交付前必须有验证或复盘证据。

### 下次避免
不要直接 full render，不要跳过参考学习、规格、预览和验证；先低成本确认方向，再投入高成本执行。

- `03_新项目创建SOP.md`
- `AGENTS.md`

### 2026-06-28｜项目：全局工作台｜主题：参考优先、预览优先和知识同步

### 问题
近期在视频生产和界面设计中出现过效果不稳定、直接进入最终实现、参考学习不足、需要反复调教的问题。

### 原因
缺少四个前置闸门：能力发现、一手资料学习、低成本预览 proof、项目不变量确认；同时跨电脑同步规则也不够清晰。

### 解决方案
明确全局规则：
- 复杂任务先查本地能力，再查一手资料。
- 风格敏感任务先预览，再完整实现。
- 跨电脑同步只同步沉淀层，不同步项目层。
- 全局规则复盘直接更新根目录 md，不新建项目。

### 下次避免
先查现成能力、先参考、先预览、先锁不变量，再开发、渲染或部署。结构不确定时先确认，不擅自扩展目录体系。

### 2026-06-29｜项目：Project_03_fuye｜主题：第一版不要追求大而全

### 问题
在调研闲鱼自动化项目时，一开始想把多账号矩阵、销售分析、选品分析、卡密仓库、下载站、飞书通知、微信 Bot 等所有功能都塞进第一版。

### 原因
对“最小可用”的理解不够深刻，把未来阶段功能和第一阶段必须功能混在了一起，没有明确边界意识。

### 造成的影响
- 第一版复杂度飙升，可能还没跑起来就放弃
- 无法快速验证核心假设
- 技术债和维护成本一开始就过高

### 解决方案
明确第一版只做 5 件事：
1. 多账号绑定和状态监控
2. 商品级自动回复
3. 虚拟商品自动发货
4. 实物商品订单推送到飞书/微信 Bot
5. 一次性取件码系统

### 下次避免
- 第一版启动前先写清楚“只做哪 5 件事”。
- 再写清楚“暂时不做哪些事”。
- 每个阶段都要有停止条件，完成后再进入下一阶段。

### 做法 001：7维度技术选型评估法

日期：2026-06-29  
关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
对开源项目进行系统化选型评估，按 7 个维度完整覆盖：
1. 需求拆解
2. 总体判断
3. 逐个项目分析
4. 关键痛点深度分析
5. 核心功能方案建议
6. 四阶段实施路线
7. 明确第一版系统边界

为什么效果好：
- 避免“功能多就选它”的盲目选型。
- 每个维度都有具体产出，而不是空泛评价。
- 边界定义避免第一版陷入大而全陷阱。

适用场景：
- 开源项目技术选型
- 复杂系统第一版方案设计
- 需求不明确时的方案调研

复用步骤：
1. 先列需求矩阵。
2. 给候选项目排序并写清理由。
3. 每个项目按“定位→能解决→不能解决→需要二开→适配度”分析。
4. 横向对比核心痛点。
5. 给出实施路线图。
6. 明确边界：做什么、不做什么。

注意事项：
- 一定要写“不能解决”和“需要二开”。
- 商业许可、平台规则、风控风险也要一起评估。

是否需要更新 AGENTS.md 或 06_工作区提示词.md：建议新增技术选型模板到 `06_工作区提示词.md`

### 做法 002：副业自动化边界设计哲学

日期：2026-06-29  
关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
副业自动化系统按风险分层设计边界：
- 低风险、高重复 → 全自动
- 中风险、偶尔发生 → 提醒人工
- 高风险、涉及钱和规则 → 必须人工

为什么效果好：
- 把自动化定位成帮手，而不是替代者。
- 减少重复劳动，同时保留关键风险控制权。

适用场景：
- 电商、支付、平台规则相关自动化系统
- 副业自动化工具设计

复用步骤：
1. 先列全业务场景。
2. 按风险程度评分。
3. 低风险全自动。
4. 中风险提醒人工确认。
5. 高风险只发消息不自动执行。

注意事项：
- 不要为了追求“全自动”而把高风险场景也自动化。

是否需要更新 AGENTS.md 或 06_工作区提示词.md：建议加入 `AGENTS.md` 的安全要求部分

### 做法 003：一次性取件码状态机设计

日期：2026-06-29  
关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
资源资料类虚拟商品，不直接发长期网盘链接，而是发一次性取件码，用状态机控制全流程：
- `unused`
- `locked`
- `used`
- `expired`
- `refunded`
- `disabled`

并配套 10 字段数据表设计、按商品类型区分下载/失效规则。

为什么效果好：
- 防止资源链接长期外传。
- 取件码与订单、买家绑定，可追溯。
- 退款后可立即作废。

适用场景：
- 虚拟资料商品发货系统
- 付费下载站、知识付费平台

复用步骤：
1. 独立一个取件码服务。
2. 设计状态机与数据表。
3. 发货时调用 API 返回链接。
4. 提取时校验状态。
5. 退款回调改为 `refunded`。

注意事项：
- 真实资源建议放对象存储并用签名 URL。
- 补发也走同样流程，不直接发原始链接。

是否需要更新 AGENTS.md 或 06_工作区提示词.md：建议作为通用设计模式沉淀

### 做法 004：四阶段系统分层实施路线

日期：2026-06-29  
关联项目：Project_03_fuye（闲鱼多账号托管调研）

做法描述：
任何复杂系统都按四阶段渐进实施：
1. 最小可用系统
2. 按业务二开
3. 稳定性和规模化
4. 数据分析和运营

为什么效果好：
- 每个阶段都有明确停止条件。
- 阶段 1 就能看到正反馈，不容易半途而废。
- 后续阶段根据真实发展决定，不会过度设计。

适用场景：
- 从零开始的软件项目
- 副业项目从 0 到 1
- 开源项目二开定制

复用步骤：
1. 先定义阶段 1 的最小可用边界。
2. 跑通后再列阶段 2 清单。
3. 每进入下一阶段前，确认上一阶段已稳定。
4. 永远不要跨阶段做未来功能。

注意事项：
- 最容易犯的错就是把阶段 4 的功能拿到阶段 1 来做。

是否需要更新 AGENTS.md 或 06_工作区提示词.md：建议加入 `03_新项目创建SOP.md`
