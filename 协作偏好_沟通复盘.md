# 协作偏好_沟通复盘

> 本文档记录跨项目、跨电脑都稳定适用的协作偏好、沟通习惯和验收方式。它不记录单个项目的细节，而只沉淀会长期影响工作方式的内容。

## 一、这次沟通容易变累的原因

1. 没有先把用户的核心目标固化成稳定规则，导致同类问题反复出现。
2. 只响应当前小改动，缺少把需求归并成长期可复用规则的动作。
3. 对“其他不变”的边界执行不够严格。
4. 结构归属不明确时，没有先问用户，而是擅自做了目录和同步判断。
5. 验收口径没有持续前置，导致用户要反复纠偏。

## 二、用户稳定偏好

- 喜欢直接执行，不喜欢只给方案或让用户反复确认。
- 偏好结构清晰、可执行、可复用的成果，而不是过程性废话。
- 偏好先把工作流、规则和边界想清楚，再做具体项目。
- 偏好低风险事项直接推进，高风险事项先说清楚风险、范围和影响。
- 偏好“同样的坑不要再踩第二次”，所以每次错误都应该沉淀成规则。
- 偏好沉淀成果统一收敛到 `D:\Workspace`，而不是散落在旧镜像目录或项目临时目录中。
- 明确要求：一个 skill 必须是一个单独文件夹，入口文件必须是 `SKILL.md`。
- 结构性不确定问题必须先反问，不能自作聪明做结构判断。

## 三、长期默认执行规则

1. **真源规则**
   - `D:\Workspace` 是唯一沉淀真源。
   - 远程仓库只是镜像，不是真源。

2. **成果边界规则**
   - 需要长期保留并跨电脑同步的，只有：
     - Workspace 根目录全局 md
     - `D:\Workspace\skills\`
   - `Project_*` 普通项目内容默认不同步到 GitHub。

3. **skills 规则**
   - 正式 skills 统一放在 `D:\Workspace\skills\`。
   - 一个 skill = 一个文件夹 = 一个 `SKILL.md`。
   - 不允许 `skills\xxx.md` 这种平铺式结构。
   - 项目目录里的 skill 副本不算正式真源。

4. **沟通规则**
   - 结果汇报要简洁，但要说清楚做了什么、放在哪里、怎么验证。
   - 如果是低风险、可回滚的小事，可以直接做。
   - 如果是结构、归属、删除、同步边界这类高风险判断，先问。

5. **沉淀规则**
   - 合并沉淀时，目标是形成一份最终成果，不是机械拼接多个旧文件。
   - 最终成果应结构统一、命名统一、逻辑完整、可长期维护。

## 四、默认验收口径

处理全局沉淀类任务时，默认检查：

1. 是否明确了真源与镜像的关系。
2. 是否只保留沉淀层，不把项目层误带入。
3. skills 是否全部符合目录式结构。
4. 最终文件是否像长期成果，而不是临时拼接稿。
5. 是否还有重复文件、重复目录或错误副本未清理。

## 五、后续遇到类似任务的默认做法

如果再出现“家庭电脑 / 公司电脑 / GitHub 仓库 / Workspace”之间的结构整理任务：

1. 先确认唯一真源在哪。
2. 先确认哪些属于沉淀层，哪些属于项目层。
3. 先确认最终成果结构，而不是上来就复制文件。
4. 先问清楚结构边界，再开始执行。
5. 最后输出一套长期维护的终稿，而不是保留多个中间版本。
