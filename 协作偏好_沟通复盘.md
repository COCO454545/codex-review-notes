# 协作偏好与沟通复盘

## 这次沟通变累的原因

1. 没有先把用户的核心目标固化下来：这是一个“自己和朋友都能用的刷题记忆系统”，不是单纯页面 demo。
2. 每次只响应当前小改动，缺少把需求归并成稳定产品规则，导致同类问题反复出现。
3. 对“其他不变”的边界执行得还不够明确：用户只要求改某一处时，应只改对应模块，并主动说明验证范围。
4. 没有提前沉淀用户偏好，例如：移动端优先、苹果科技风、题量不能少、错题解释要短且逐题唯一、上线后要直接同步原链接。
5. 验收口径没有一直前置，应该每次修改后固定校验题量、解释数量、线上是否更新。

## 用户稳定偏好

- 喜欢直接执行，不喜欢只给方案或让用户反复确认。
- 对题库完整性非常敏感：理论题 900 道、实操题 161 道，不能少题、乱序、混入水印。
- 错题解释要“短、准、逐题不同”，不要模板废话，不要长篇展开。
- UI 偏好 Apple 科技风：轻量、干净、移动端优先、状态清晰。
- 功能修改后要直接同步到原线上链接，除非用户明确要求只本地预览。
- 用户经常说“其他不变”，此时只处理指定问题，不顺手改无关功能。
- 需要面向真实分享使用：分享码、密码、个人错题隔离、管理统计都应保留。

## 后续默认执行规则

1. 改功能前先确认影响面：题库数据、答题交互、错题本、管理端、发布链路。
2. 修改生成源文件优先，例如 `build_combined_portal.py`，避免只改 `dist_combined/index.html` 后被重新构建覆盖。
3. 每次发布前至少校验：
   - `theory.json` 为 900 道。
   - `practical.json` 为 161 道。
   - `explanations.json` 为 1061 条且尽量唯一。
   - 生成后的 HTML JavaScript 语法通过。
4. 发布后再拉取线上页面或数据确认最新代码已经生效。
5. 最终回复要短，直接说明改了什么、链接是什么、校验了什么。
